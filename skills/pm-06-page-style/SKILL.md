# PM 06 Page Style

## Goal

Create a structured Figma JSON node document for exactly one page layout document generated by `pm-03-page-layout`.

Primary input:

```text
product/pages/**/layout.md
```

Design input:

```text
design/<设计规范名称>/
├── design-spec.md
├── tokens.json
├── figma-tokens.json
├── component-rules.json
├── component-blueprints.json
├── figma-render-rules.json
├── ai-generation-rules.json
├── consistency-rules.json
├── layout-profile-schema.json
└── page-intent-patterns.json
```

Outputs, written next to the source `layout.md`:

```text
product/pages/**/figma-nodes.json
product/pages/**/figma-node-notes.md
```

This skill does not create or edit a Figma file directly. It prepares a stable JSON contract that later AI agents or Figma tools can use to create the design in Figma.

Important scope constraint: generate only the default-state page structure. Non-default states such as empty, loading, error, success, modal-open, drawer-open, and disabled variants must not become Figma page nodes in `figma-nodes.json`. They may be recorded as excluded references in notes or metadata.

- Use `agents/gemini.md` and `agents/claude.md` as lightweight adapter prompts in Gemini or Claude environments.

## Scope Rule

Each invocation generates or updates exactly one page's Figma JSON node document.

Allowed user targets:

- One explicit `product/pages/**/layout.md` path.
- One page directory containing `layout.md`.
- One `PAGE-xxx` ID that can be matched to exactly one `product/pages/**/layout.md`.

If the user requests multiple pages or a batch, stop and ask them to use a future batch orchestration skill or provide one page target for this run.

## Consistency Profile Resolution

`pm-06` must preserve consistency across more than state groups. Before generating nodes, resolve these keys:

- `layoutKey`: normalized `使用Layout`, e.g. `客户端-PC-Web`, `管理后台-PC-Web`.
- `pageIntentKey`: selected generic `pageIntent.primary`, e.g. `auth`, `list-management`.
- `stateGroupKey`: `状态组` when present; otherwise empty.
- `designSpecKey`: selected design directory name.

Project-level profile directory:

```text
product/style-profiles/<layoutKey>/
├── layout-profile.json
├── intent-<pageIntentKey>-profile.json
└── state-<stateGroupKey>-profile.json
```

Profile rules:

1. Always load `design/<设计规范名称>/consistency-rules.json`, `layout-profile-schema.json`, and `page-intent-patterns.json` when present. If an older design spec lacks them, continue only when the required layout/intent/state profile can be derived from existing tokens and render rules; otherwise stop and ask the user to rerun `pm-05-design-spec`.
2. If `layout-profile.json` does not exist, create it from the source sitemap global layout, `使用Layout`, design layout tokens, and `figma-render-rules.json`. It locks canvas size, shell areas, navigation, page container, global padding, section gap, and responsive fallback.
3. If `intent-<pageIntentKey>-profile.json` does not exist, create it from `page-intent-patterns.json`, `component-rules.json.semanticPatterns`, `ai-generation-rules.json.pageIntentStrategies`, and the selected design tokens. It locks the generic visual structure for that intent under the current layout.
4. If the page has a `状态组` and `state-<stateGroupKey>-profile.json` does not exist, create it from the first generated page in that state group. It locks shared state-group geometry such as auth card width, method switch style, form width, control height, QR size, status header, wizard/timeline geometry, and shared component variants.
5. If profiles already exist, load them and reuse locked values. Do not re-decide locked values from the current page content.
6. A page may fill slots and change business content, but it must not change layout/profile-locked visual decisions without explicit user instruction.
7. When an existing `figma-nodes.json` has a different fingerprint, update it to the current profile unless the user explicitly requests preserving the old variant.

`figma-nodes.json` must include a top-level `consistency` object:

    },

      "status": "pass|warning|failed",

Fingerprint rules:

- Use deterministic canonical JSON for profile hash inputs: sort object keys, remove timestamps, and hash only locked style/layout fields.
- `layoutHash` covers canvas, shell, navigation, page container, global spacing, and responsive shell rules.
- `intentHash` covers intent skeleton, common regions, component variants, typography scale usage, region gaps, and forbidden regions.
- `stateGroupHash` covers shared state-group frame/card/field/tab/button/QR/table/timeline geometry and component variants.

## Sitemap Layout Merge

`pm-03` page documents intentionally focus on page body content. `pm-06` must rebuild the full page by merging the page body with the global layout from the source sitemap.

From `来源Sitemap`, read:

- `0.文档状态` for product end and device shape.
- `1.layout布局方式`.
- `1.2.区域、分组与元素` for global shell areas and elements.
- `2.1.sitemap思维导图` for navigation context.
- `2.2.页面清单` row matching `页面清单ID` plus its parent chain.

Full page composition:

```text
Figma Page
├── Global Layout Shell
│   ├── Topbar / Header
│   ├── Sidebar / BottomNav / AppNav
│   ├── Breadcrumb / Page Container
│   └── Global Actions
└── Page Content from layout.md
    ├── Page Title Area
    ├── Filter Area
    ├── Main Content
    └── Table / List / Card / Form
```

If the sitemap file is missing, stop and ask the user to restore or regenerate it with `pm-02-sitemap`.

Do not merge state-difference content into the page shell. Global layout shell is merged because it is part of the complete default page; non-default page states are not merged because they are alternate renderings.

## Page Intent Recognition

Before generating any Figma nodes, classify the page into a generic `pageIntent`. This is mandatory and must be written to `figma-nodes.json`.

Allowed intent values:

- `auth`
- `dashboard`
- `list-management`
- `detail`
- `form-create-edit`
- `checkout-payment`
- `wizard-flow`
- `profile-settings`
- `message-center`
- `document-viewer`
- `empty-or-state-page`
- `custom`

Use multiple evidence sources, not only the page name:

- `页面清单ID`, `页面/模块`, `页面类型`, `状态组`, `用户角色`, `核心场景`, and parent chain from the source sitemap.
- `layout.md` sections `1.1`, `1.3.1`, `1.3.2`, `1.4`, `1.5`, and default-state mock data.
- `component-rules.json.semanticPatterns`.
- `ai-generation-rules.json.pageIntentStrategies`.

Write the decision as:

```json
{
  "pageIntent": {
    "primary": "auth",
    "confidence": "high",
    "evidence": [],
    "selectedPattern": "component-rules.semanticPatterns.auth",
    "strategy": "ai-generation-rules.pageIntentStrategies.auth"
  }
}
```

Intent classification must be product-agnostic. Do not hard-code current project page names as rules. For example, identify authentication by the presence of identity input, verification/password input, agreement, auth method switch, and submit action; identify list management by filter controls, collection/table/list, toolbar, row actions, and pagination.

## Semantic Structure Strategies

After `pageIntent` is chosen, generate structure according to the selected intent strategy. The strategy may be adapted to the current design spec, but the semantic layout must remain appropriate.

Generic structure expectations:

- `auth`: global shell if applicable, auth panel, auth method switch, identity input, verification/password/SSO control, agreement or policy link when present, primary submit, auxiliary links. Do not invent a data table unless the source explicitly states an auth audit table is default-visible.
- `dashboard`: page header, metric/summary cards, task or alert list, chart or trend area, shortcuts.
- `list-management`: page header, filter/search area, toolbar/batch actions, table/list/card collection, row actions, pagination.
- `detail`: page header, summary/status area, description fields, related records, timeline/log, action group.
- `form-create-edit`: page header, grouped form sections, validation hints, upload or rich controls when present, sticky or bottom action bar.
- `checkout-payment`: order summary, price breakdown, payment method, agreement/risk hints, primary payment action, payment-status reference.
- `wizard-flow`: steps/progress, current step body, previous/next/submit actions, step status.
- `profile-settings`: settings navigation, profile/security/preference sections, save actions.
- `message-center`: filters, message list, read/unread state, preview/detail area.
- `document-viewer`: toolbar, preview/content panel, metadata/actions panel.
- `empty-or-state-page`: current default state content only, with explicit note that sibling non-default states are excluded.
- `custom`: preserve the structure from `1.4` and `1.5` while recording why no stronger intent matched.

Do not coerce every page into a table. Table generation is appropriate only when the intent or source structure says the primary content is a collection/list/table.

## Semantic Validation

Before writing the final JSON, compare `pageIntent` against the extracted `1.5.页面元素清单` and generated node tree.

Write a `semanticValidation` object:

```json
{
  "status": "pass",
  "conflicts": [],
  "warnings": [],
  "criticalMissing": [],
  "autoFixes": [],
  "semanticOwnershipAudit": {},
  "duplicateAudit": {},
  "layoutBudgetAudit": {}
}
```

Conflict examples:

- `auth` with default-visible `TABLE` as the primary content.
- `list-management` without any table/list/card collection.
- `form-create-edit` without submit/cancel/save actions.
- `checkout-payment` without amount, order summary, or payment action.
- `detail` without detail fields or summary/status content.
- `wizard-flow` without current step content or navigation action.

Handling:

- If a severe conflict means the generated design would be semantically wrong, mark the page task as `Failed` rather than producing misleading `figma-nodes.json`.
- If the conflict is repairable from `1.3.1`, `1.4`, sitemap context, or design semantic patterns, fix the node tree using the page-intent strategy and record the repair in `semanticValidation.autoFixes` and `figma-node-notes.md`.
- Never silently pass schema-valid but semantically wrong JSON.

### Semantic Ownership, Duplicate, And Budget Audits

`pm-06` must treat `layout.md` as product semantics and compile it into render-safe JSON. Before writing `figma-nodes.json`, run these audits:

1. `semanticOwnershipAudit`
   - Classify every default-visible node as `Global Shell`, `Page Body`, `Overlay/State`, or `Mock Data`.
   - Global shell nodes may come only from `来源Sitemap` merge. Page-body nodes must not duplicate global topbar/sidebar/footer/current-navigation semantics.
   - If a page-body node repeats global text intentionally, record distinct roles; otherwise remove it from page-body generation or fail with `duplicate-global-shell-semantics`.
2. `duplicateAudit`
   - Build a list of visible text strings, interactive elements, and `actionIntent` values.
   - Allow repeated text only when roles differ, such as shell product name vs page hero title.
   - Fail when the same page body has two default-visible primary actions with the same `actionIntent`.
   - Warn when the same helper link or navigation destination appears twice in the same region.
3. `layoutBudgetAudit`
   - Load `figma-render-rules.layoutRecipes`, `figma-render-rules.measurementRules`, `consistency-rules.layoutBudgetRules`, and matching `page-intent-patterns.layoutBudgets` when present.
   - Compute available root/content height from canvas minus reserved shell regions and page padding.
   - Estimate major container heights from fixed components, text policies, gap, padding, and slot content.
   - If content exceeds the budget, apply the design's compression order in JSON constraints: reduce noncritical gaps, cap optional slot height, mark scroll containers, or shrink allowed media/QR/preview sizes.
   - If no legal compression can fit the default state, set `semanticValidation.status = "failed"` and do not emit a misleading render-ready JSON.

Audit results must be copied into both `semanticValidation` and `renderSpec.qaExpectations`; batch skills rely on these fields.

## Render-Ready Contract

`figma-nodes.json` must include a `renderSpec` object. This is the binding contract between `pm-06` and `pm-08`; it must be specific enough that a Figma importer does not need to guess layout, component structure, or overflow behavior from prose.

Required structure:

```json
{
  "renderer": {
    "sourceTree": "figmaDocument.children",
    "requiresComponentAdapters": true,
    "requiresTwoPassLayout": true,
    "preferredRenderRules": "design/<设计规范名称>/figma-render-rules.json"
  },
  "canvas": {
    "primaryFrameId": "frame-primary",
    "device": "{{primary-device}}",
    "width": 1440,
    "minHeight": 900,
    "layoutMode": "VERTICAL"
  },
  "layoutSafety": {
    "noOverlap": true,
    "noTextOverflow": true,
    "avoidClipping": true,
    "minClickableTarget": 32
  },
  "layoutBudget": {
    "availableContentHeight": 0,
    "estimatedContentHeight": 0,
    "maxPrimaryPanelHeight": null,
    "compressionOrder": [],
    "scrollContainers": [],
    "status": "pass|warning|failed"
  },
  "componentInstances": [],
  "textPolicies": [],
  "constraints": [],
  "slotContracts": [],
  "duplicateAudit": [],
  "qaExpectations": {}
}
```

Rules:

- Every `COMPONENT_INSTANCE` in `figmaDocument` must have a matching `renderSpec.componentInstances` entry with `sourceNodeId`, `component`, `adapter`, `variant`, `props`, `slots`, `sizing`, and `textPolicy`.
- Every text-heavy node must have a matching `renderSpec.textPolicies` entry defining wrap/truncate behavior, max lines, minimum width, and whether the parent may grow vertically.
- Every major frame, section, table, form row, card, and shell container must have a matching `renderSpec.constraints` entry defining width mode, height mode, min/max values, alignment, gap, and padding source.
- `renderSpec.layoutBudget` must record available content dimensions, estimated content dimensions, compression actions, scroll containers, and whether the page is safe to restore.
- `renderSpec.slotContracts` must record shared and variable slots from page-intent/state-group profiles, including which slot may vary by page/state and which dimensions are locked.
- `renderSpec.duplicateAudit` must record duplicate text and duplicate action-intent findings. Blocking duplicate primary actions must fail generation.
- `renderSpec.qaExpectations.keyRegions` must list expected semantic regions from `pageIntent`, such as auth panel, submit action, filter bar, data collection, detail summary, or payment action.
- `renderSpec.qaExpectations.forbiddenRegions` must list structures that must not appear for the selected intent, such as default-visible primary data tables on an auth page unless explicitly sourced.
- `renderSpec.qaExpectations` must include required post-render checks: no shell collision, no root clipping, no incoherent overlap, no duplicate primary action intent, no text overflow, and no unexpected non-default state nodes.
- Use `component-blueprints.json` for the adapter and prop contract. Use `figma-render-rules.json` for overflow, two-pass layout, append, and QA expectations.
- If the source page cannot be expressed with render-safe constraints, fail the run or write `semanticValidation.status = "failed"` instead of emitting a vague JSON tree.

| 输入框 | `COMPONENT_INSTANCE` with `component = Input` |

If an element type is unknown, create a generic `FRAME` or `TEXT` node and record the fallback in `assumptions`.

Default-state filtering:

- Include elements whose `状态/数据分类` is empty, `默认状态`, `默认状态数据集`, or includes default visible page content.
- Exclude elements whose `状态/数据分类` is `空状态`, `加载状态`, `错误状态`, `成功状态`, `状态差异数据集`, or any non-default state.
- Exclude rows where `是否状态差异元素 = 是`.
- Exclude modal, drawer, dialog, toast, alert, empty, loading, and error elements unless `layout.md` explicitly marks them as default-state visible content.
- Use excluded items only to populate `excludedStates` and `figma-node-notes.md`.

## Layout Generation Rules

1. Resolve design spec and consistency profiles before creating the node tree.
2. Create a page root frame from `layout-profile.json`; do not re-infer canvas, shell, or page container geometry from the current page.
3. Add the global layout shell from the sitemap using the locked layout profile.
4. Select a `pageIntent` and apply `intent-<pageIntent>.json` before mapping individual elements.
5. If a `状态组` exists, apply `state-<STATE>.json` to lock shared same-state-group geometry and component variants.
6. Mount page body content inside the correct content container from the global shell.
7. Use default-state rows in `1.5.页面元素清单` as the structured element source, but validate them against `pageIntent`. If `1.5` is semantically inconsistent, repair from `1.3.1` and `1.4` or fail the page.
8. Use `1.4.默认状态页面结构图` to preserve grouping and hierarchy when it matches the list and the selected page intent.
9. Use only default-state rows in `2.Mock数据` to populate sample text, table rows, badges, and placeholders.
10. Do not create separate nodes for empty, loading, error, success, modal-open, drawer-open, or other non-default states.
11. Record excluded state rows in `excludedStates` and explain the exclusion in `figma-node-notes.md`.
12. Generate `renderSpec` after the semantic node tree is built, binding every component instance, major container, and text-heavy node to concrete adapter, sizing, overflow rules, and consistency profile references.
13. Generate and validate `renderSpec.layoutBudget`, `renderSpec.slotContracts`, and `renderSpec.duplicateAudit` before writing JSON.
14. Before writing JSON, check the generated tree against `figma-render-rules.json.semanticKeyRegions`, `figma-render-rules.json.qaRules`, `figma-render-rules.layoutRecipes`, `consistency-rules.layoutBudgetRules`, `consistency-rules.semanticUniquenessRules`, and the loaded consistency profiles; repair or fail rather than emitting a structurally valid but visually unsafe or inconsistent document.

Consistency-specific generation rules:

- Same `使用Layout` pages must share shell structure, canvas width/min-height, page padding, global navigation dimensions, content max width, section gaps, and responsive fallback strategy.
- Same `使用Layout + pageIntent` pages must share major intent structure and visual proportions. For example, all auth pages under one layout must share auth panel direction, brand panel width, card width, card padding, tabs style, and auxiliary link placement unless the profile explicitly defines alternatives.
- Same `状态组` pages must share the state-group skeleton. For example, login method states may change active tab and credential/SSO slot content, but not card width, tabs gap, brand typography, button height, or shared shell dimensions.
- If a current page requires a different geometry for legitimate content reasons, record it as `consistency.allowedPageOverrides` and ensure it does not mutate the profile for sibling pages.
- Same `使用Layout + pageIntent + stateGroup` pages must share slot contracts and layout budgets. State/page-specific content may fill a variable slot differently, but it must not force shared regions outside safe bounds unless recorded as an allowed override and marked for manual review.

When `1.4` and `1.5` conflict, do not blindly prefer one side. Resolve using `pageIntent` and evidence quality:

- Prefer `1.5` when it is semantically consistent with `pageIntent`.
- Prefer `1.4` or `1.3.1` when `1.5` appears mechanically generic or contradicts the page intent.
- Record the decision and evidence in `semanticValidation` and `figma-node-notes.md`.

## Responsive Variants

Always include `responsiveVariants`.

Default frame width:

- `PC Web`: `1440`
- `Mobile App`: `390`
- `Mobile Web`: `390`
- `Tablet`: `768` or `834`
- `PC Desktop App`: `1600`

If `使用Layout` implies PC Web, generate the desktop variant as primary and include mobile fallback rules. If it implies Mobile App, generate mobile as primary and include tablet/desktop adaptation notes when the design spec supports them.

Table handling:

- Desktop: render full table with toolbar and pagination.
- Tablet: keep core columns and detail panel when needed.
- Mobile: convert table rows to CardList items using `component-rules.json.responsiveFallbacks.TableToCardList`.
  Record this conversion in `renderSpec.constraints` and `renderSpec.componentInstances` so the importer can create a stable fallback.
