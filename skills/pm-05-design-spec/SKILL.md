# PM 05 Design Spec

## Goal

Create or update one complete design specification from the user's natural-language design request.

Output location:

```text
design/<设计规范名称>/
├── design-spec.md
├── index.html
├── tokens.json
├── figma-tokens.json
├── component-rules.json
├── component-blueprints.json
├── figma-render-rules.json
├── ai-generation-rules.json
├── consistency-rules.json
├── layout-profile-schema.json
└── page-intent-patterns.json
```

The output must be useful for downstream AI page generation and Figma JSON node generation. It must not be a vague brand description. Every generated design spec must contain concrete tokens, component rules, responsive layout rules, and explicit constraints for future AI-generated pages. Every generated JSON file must be valid JSON and stable enough for later skills to read without parsing Markdown.

Write `design-spec.md` in Chinese. The HTML preview can use Chinese interface copy by default unless the user requests another language.

## Cross-Model Compatibility

Support GPT/Codex, Gemini, Claude, and other capable AI assistants through the same workflow and output schema.

- Treat `SKILL.md` as the canonical process definition.
- Treat `assets/design-spec-template.md` as the canonical Markdown schema.
- Keep `agents/openai.yaml` only as GPT/Codex UI metadata; it is not the source of truth.
- Use `agents/gemini.md` and `agents/claude.md` as lightweight adapter prompts in Gemini or Claude environments.
- Never create model-specific variants of generated design specs.

## Inputs

Required:

- The user's natural-language design description.

Optional context:

- Product notes in the current request.
- Existing documents under `product/` and `docs/`, only when they help infer product type, audience, domain, device shape, or interface density.
- Existing `design/<设计规范名称>/design-spec.md`, `index.html`, and JSON files, when updating a previously generated spec.

If the user description is short or abstract, do not stop to ask unless a missing decision would create incompatible output. Make a reasonable design-system assumption and record it in `12.待确认与假设`.

## Output Naming

Generate a concise, stable directory name from the user's description.

Rules:

1. Extract 2-5 meaningful keywords from the style, domain, and device description.
2. Convert English words to lowercase kebab-case.
3. For Chinese-only descriptions, translate key concepts to short English slugs.
4. Suffix with `design-spec` only when the name would otherwise be too generic.
5. Avoid dates, random IDs, and model names.

Examples:

- `创建iOS风格设计` -> `design/ios-style-design-spec/`
- `高级金融App风格` -> `design/premium-finance-app/`
- `企业后台现代简洁风格` -> `design/modern-enterprise-dashboard/`
- `多端响应式医疗平台设计` -> `design/responsive-healthcare-platform/`

If the target directory already exists, update the existing files in place. Preserve user edits unless newer user instructions clearly supersede them.

## Generation Workflow

1. Parse the user's description.
   - Identify style keywords, product domain, audience, device forms, interaction density, and emotional tone.
   - If the domain is missing, use a general product interface context.
   - If devices are missing, support mobile app, mobile web, tablet, PC web, and PC desktop app by default.

2. Infer a complete design direction.
   - Convert abstract words into concrete visual decisions.
   - Define colors, typography, spacing, radii, shadows, borders, layout, components, states, and responsive behavior.
   - Record assumptions rather than asking for confirmation.

3. Create `design-spec.md`.
   - Open `assets/design-spec-template.md`.
   - Preserve every heading and section order from the template.
   - Fill every section with concrete, implementation-ready guidance.
   - Ensure the document includes AI page-generation constraints.

4. Create machine-readable JSON files.
   - Create `tokens.json` for general design tokens.
   - Create `figma-tokens.json` for Figma-friendly variable names and typed values.
   - Create `component-rules.json` for component sizing, style, state, and responsive fallback rules.
   - Create `component-blueprints.json` for editable Figma node structures and component fallback blueprints.
   - Create `figma-render-rules.json` for importer layout, overflow, adapter, append, and visual QA rules.
   - Create `ai-generation-rules.json` for hard page-generation constraints and layout selection rules.
   - Create `consistency-rules.json` for cross-page consistency priority, profile resolution, fingerprint rules, and drift handling.
   - Create `layout-profile-schema.json` for project-level layout, intent, and state-group profile schemas.
   - Create `page-intent-patterns.json` for reusable page-intent visual structure patterns that downstream skills can lock and reuse.
   - Keep token names and values synchronized with `design-spec.md` and `index.html`.

5. Create `index.html`.
   - Use the same design tokens as `design-spec.md`.
   - Prefer reading token values from the JSON plan before writing CSS variables.
   - Build a self-contained visual preview page.
   - The HTML must demonstrate the design system, not just describe it.

6. Validate output.
   - Confirm all required files exist under `design/<设计规范名称>/`.
   - Confirm all JSON files parse as valid JSON.
   - Confirm Markdown, JSON, and HTML tokens match.
   - Confirm the HTML includes color palettes, typography, component examples, visual hierarchy, and multi-device sample screens.

## Abstract Prompt Expansion

When the user gives a short prompt, expand it into a complete design system.

For `iOS` or Apple-like prompts:

- Use calm, native, touch-first, layered, high-legibility design.
- Prefer system blue, neutral surfaces, translucent panels, large titles, grouped lists, segmented controls, bottom tabs, sheet-like panels, and subtle separators.
- Avoid Android Material-specific visual language unless requested.
- Mobile app should be primary; tablet and desktop should adapt into split-view or sidebar layouts.

For enterprise/admin prompts:

- Use restrained colors, high information clarity, compact controls, predictable navigation, tables, filters, dense forms, and clear operational states.
- Avoid oversized marketing-style heroes and decorative card-heavy layouts.

For premium/finance prompts:

- Use high contrast, disciplined spacing, refined neutrals, confident accent colors, numeric typography, trust-building status colors, and conservative motion.
- Avoid playful illustration-heavy design unless requested.

For tech/SaaS prompts:

- Use clear hierarchy, modular panels, data visualization surfaces, crisp controls, and efficient responsive layouts.
- Avoid one-note purple/blue gradient visual systems unless the user explicitly asks for that palette.

For consumer/lifestyle prompts:

- Use more expressive imagery, stronger emotional tone, lighter data density, and clearer content cards.
- Still keep tokens and components concrete enough for implementation.

## Responsive Coverage

Every generated design spec must support these device forms unless the user explicitly narrows scope:

- Mobile App: native-like touch layout, safe areas, bottom navigation or task-specific primary actions.
- Mobile Web: browser-safe stacking, reduced chrome, accessible tap targets, no horizontal scrolling.
- Tablet: two-column or master-detail layouts where appropriate.
- PC Web: sidebar/topbar, content grid, tables/forms, hover/focus states.
- PC Desktop App: denser multi-pane workspace, persistent navigation, keyboard-friendly controls.

Define breakpoints in `design-spec.md` and implement matching responsive CSS in `index.html`.

Suggested default breakpoints:

```text
Mobile: 0-599px
Tablet: 600-1023px
Desktop: 1024-1439px
Wide Desktop: 1440px+
```

## Markdown Template Rules

Before writing `design-spec.md`, open `assets/design-spec-template.md` and use it as the fixed skeleton.

Rules:

- Preserve all top-level headings and section order.
- Fill all placeholder labels with generated content.
- Keep token tables concrete. Include names, values, and usage.
- Include explicit "do" and "do not" constraints for AI page generation.
- Use examples that match the inferred product context.
- Record assumptions in `12.待确认与假设`.

## HTML Preview Requirements

`index.html` is a visual design-system board plus product interface sample. It must let the user inspect the generated design direction directly in a browser.

Technical rules:

- Single self-contained HTML file.
- CSS inside `<style>`.
- No build step.
- Avoid external CDN dependencies unless the user explicitly permits them.
- Use CSS variables for every design token.
- Tokens in HTML must match `design-spec.md`, `tokens.json`, and `figma-tokens.json`.
- Must be responsive at mobile, tablet, desktop, and wide desktop widths.
- Must not be a marketing landing page.
- Must not be a text-only documentation page.

Required HTML sections:

1. Overview
   - Design name, original prompt summary, style keywords, suitable devices, and one-sentence design positioning.

2. Color System
   - Visual swatches for primary, secondary, accent, background, surface, text primary, text secondary, border, success, warning, error, and info.
   - Each swatch must show color name, value, usage, and readable text example.

3. Typography
   - Visual examples for H1, H2, H3, body, secondary text, labels, captions, and numeric text.
   - Show size, weight, and line-height notes.

4. Spacing / Radius / Shadow
   - Show spacing scale, radius samples, elevation or border-layer samples, and surface layering.

5. Components
   - Show primary, secondary, ghost, danger, disabled, and loading buttons.
   - Show input states: default, focus, error, disabled.
   - Show tabs or segmented control, switch, checkbox, badge/tag, alert/toast, card/list item, table row, modal/drawer/sheet sample.

6. Mobile App Sample
   - A narrow app-like frame with navigation, content, controls, list/card elements, and bottom navigation or fixed primary action.

7. PC Web Sample
   - A desktop web layout with navigation, main content, cards/metrics, table/list, filters, and actions.

8. PC Desktop App / Wide Workspace Sample
   - A multi-pane desktop-app-like layout, such as sidebar, middle list, and right detail panel.

9. Responsive Notes
   - Summarize how the design changes across breakpoints.

HTML visual quality rules:

- Use the generated visual style throughout all examples.
- Do not rely on browser defaults for components.
- Avoid single-hue monotony unless the user specifically requests monochrome.
- Keep text inside UI elements readable and non-overlapping.
- Use stable dimensions for cards, controls, sample screens, and preview frames.
- Use real product-like interface examples rather than placeholder lorem ipsum.

## JSON Output Requirements

Create nine valid JSON files for downstream skills. These files are machine-readable contracts; keep keys stable, concise, and predictable.

### `tokens.json`

General design tokens for AI page generation and frontend generation.

Required top-level keys:

```json
{
  "meta": {},
  "color": {},
  "typography": {},
  "spacing": {},
  "radius": {},
  "shadow": {},
  "border": {},
  "breakpoint": {},
  "layout": {},
  "zIndex": {},
  "motion": {}
}
```

Token rules:

- Use numeric values for Figma-friendly dimensions where possible, e.g. `{ "value": 16, "unit": "px" }`.
- Use hex colors for stable color values. If an HTML token uses rgba, provide an equivalent named shadow/border token separately.
- Include `usage` on tokens that downstream layout generation might choose from.
- Use token names that can be referenced from component rules, such as `color.primary`, `spacing.4`, `radius.md`.

### `figma-tokens.json`

Figma-oriented variables and node-generation defaults.

Required top-level keys:

```json
{
  "meta": {},
  "variables": {},
  "textStyles": {},
  "effectStyles": {},
  "layoutDefaults": {}
}
```

Rules:

- Variable keys should use slash paths, e.g. `color/primary`, `spacing/4`, `radius/md`.
- Variable types must be one of `COLOR`, `FLOAT`, `STRING`, or `BOOLEAN`.
- Color variables use hex values.
- Text styles must include `fontFamily`, `fontSize`, `fontWeight`, and `lineHeight`.
- Layout defaults should define common Figma auto-layout values for page, card, form row, table row, mobile screen, and desktop shell when relevant.

### `component-rules.json`

Component-level rules for mapping page descriptions into Figma JSON nodes.

Required top-level keys:

```json
{
  "meta": {},
  "components": {},
  "semanticPatterns": {},
  "responsiveFallbacks": {},
  "stateMapping": {}
}
```

Rules:

- Include rules for at least Button, Input, Select, Checkbox, Switch, Tabs or SegmentedControl, Navigation, Card, Table, List, Modal or Drawer, Toast or Alert, Empty, Loading, and Error.
- Include `semanticPatterns` for common page intents. These patterns tell downstream skills which component families belong to each page type, without tying the design spec to a specific product.
- Component rules should reference token paths instead of duplicating raw values when possible.
- Include default sizes, padding, radius, fill, stroke, typography, states, and responsive fallback.
- For admin/table-heavy specs, define how Table converts to mobile CardList.

Required generic `semanticPatterns` keys unless the user explicitly narrows scope:

```json
{
  "auth": ["AuthPanel", "Tabs", "Input", "VerificationCode", "Checkbox", "Button", "AuxiliaryLink"],
  "dashboard": ["MetricCard", "TaskList", "Chart", "ShortcutAction", "Alert"],
  "list-management": ["PageHeader", "FilterBar", "Toolbar", "Table", "Pagination", "BatchAction"],
  "detail": ["PageHeader", "DescriptionList", "Timeline", "StatusTag", "ActionGroup"],
  "form-create-edit": ["FormSection", "Input", "Select", "Upload", "ValidationMessage", "ActionBar"],
  "checkout-payment": ["OrderSummary", "PriceBreakdown", "PaymentMethod", "Agreement", "PrimaryAction", "PaymentStatus"],
  "wizard-flow": ["Steps", "CurrentStepPanel", "FormSection", "PrevNextAction", "ProgressStatus"],
  "profile-settings": ["SettingsNav", "ProfileForm", "SecurityPanel", "PreferenceSwitch"],
  "message-center": ["MessageFilter", "MessageList", "ReadState", "MessageDetail"],
  "document-viewer": ["DocumentToolbar", "PreviewPanel", "MetadataPanel", "DownloadAction"],
  "custom": ["PageHeader", "ContentSection", "ActionGroup"]
}
```

Each pattern may add product-specific components, but must not remove the generic intent key itself. Downstream skills use these keys for page-intent mapping and semantic quality checks.

### `component-blueprints.json`

Editable Figma node blueprints for downstream Figma restoration.

Required top-level keys:

```json
{
  "meta": {},
  "componentBlueprints": {},
  "semanticBlueprints": {},
  "slotRules": {},
  "actionRules": {},
  "textOverflowRules": {},
  "layoutSafety": {}
}
```

Rules:

- Include blueprints for at least Button, Input, Select, Checkbox, Switch, Tabs or SegmentedControl, Badge or Tag, Steps, Upload, Table, Card, FormSection, Topbar, Sidebar, Footer, Modal or Drawer.
- Each `componentBlueprints.*` entry must define editable node structure, layout direction, sizing mode, min/max dimensions, padding, gap, radius, fill, stroke, text style, icon slot behavior, and supported variants.
- Each component blueprint must expose a stable `props` contract, such as `label`, `placeholder`, `value`, `state`, `size`, `variant`, `icon`, `items`, `columns`, or `rows`, so `pm-06` and `pm-08` do not infer component shape from display text.
- Include `textOverflowRules` for label wrapping, placeholder truncation, table cell wrapping, max lines, and minimum text container widths.
- Include `layoutSafety` defaults for no-overlap spacing, container min-height, control min-width, tap/click targets, and fixed-width columns.
- Include `actionRules` that define how buttons, links, row actions, navigation actions, and icon buttons expose `actionIntent`, `priority`, `isPrimary`, and duplicate-action handling. Rules must be product-agnostic and must not allow two default-visible primary actions with the same intent inside one page body.
- Include `slotRules` for generic intent slots such as auth credential slot, table toolbar slot, detail action slot, wizard navigation slot, checkout payment slot, status message slot, and settings panel slot. Each slot rule must define allowed child component families, min/max sizing, and whether the slot may scroll.
- Keep blueprints product-agnostic. They may express the design style, but must not mention current project page names or business entities.

### `figma-render-rules.json`

Renderer-level rules for turning `pm-06` JSON into visually stable Figma nodes.

Required top-level keys:

```json
{
  "meta": {},
  "renderDefaults": {},
  "layoutRecipes": {},
  "layoutEngine": {},
  "componentAdapters": {},
  "overflowRules": {},
  "measurementRules": {},
  "semanticKeyRegions": {},
  "appendRules": {},
  "qaRules": {},
  "failurePolicy": {}
}
```

Rules:

- `renderDefaults` must define frame width, page padding, shell spacing, section spacing, background behavior, and default auto-layout sizing for supported device forms.
- `layoutRecipes` must define reusable, device-aware layout recipes for common page intents such as auth, dashboard, list-management, detail, form-create-edit, checkout-payment, wizard-flow, status, message-center, document-viewer, and profile-settings. Each recipe must define safe area, shell budget, content max/min dimensions, default gaps, density modes, scroll containers, and compression order.
- `layoutEngine` must require deterministic two-pass layout: create parent frames and component instances first, then apply resolved child sizing, min/max constraints, and alignment after children are mounted.
- `componentAdapters` must map the design components to Figma restoration adapters. Include at least Button, Input, Select, Checkbox, Switch, Tabs or SegmentedControl, Badge or Tag, Steps, Upload, Table, Card, Topbar, Sidebar, Footer, Modal or Drawer.
- `overflowRules` must define hard rules for text wrapping, table cell clipping avoidance, form row wrapping, vertical scrolling containers, and when to increase container height instead of clipping content.
- `measurementRules` must define how downstream renderers compute expected content height before final placement, including handling for `hug`, `fill`, min/max height, fixed shell regions, footer/header avoidance, and fallback scroll areas.
- `semanticKeyRegions` must list required and forbidden visible regions per generic page intent, for example `auth` requires auth panel and submit action, while default-visible primary tables are forbidden unless source content explicitly says so.
- `appendRules` must describe how Figma importers create comparison frames without overwriting existing same-name nodes.
- `qaRules` must require screenshot or structure inspection after restore, checking no text overflow, no incoherent overlap, no root-frame clipping, no shell/content collision, no duplicate primary action intent, meaningful visual hierarchy, expected key regions, and no generic template pollution.
- `failurePolicy` must require the downstream skill to fail or mark manual-review when it cannot satisfy layout safety, rather than silently creating an unattractive or stacked design.

### `ai-generation-rules.json`

Hard constraints for downstream AI agents.

Required top-level keys:

```json
{
  "meta": {},
  "styleIntent": {},
  "deviceStrategies": {},
  "layoutPatterns": {},
  "pageIntentStrategies": {},
  "semanticOwnershipRules": {},
  "duplicatePrevention": {},
  "requiredRules": [],
  "forbiddenRules": [],
  "componentSelection": {},
  "figmaNodeGenerationHints": {},
  "qualityChecklist": []
}
```

Rules:

- Store direct, imperative rules that downstream skills can apply without re-reading Markdown.
- Include device-specific layout strategy for mobile app, mobile web, tablet, PC web, and PC desktop app.
- Include `pageIntentStrategies` that maps generic intents such as `auth`, `list-management`, `detail`, `form-create-edit`, `checkout-payment`, `wizard-flow`, and `dashboard` to layout strategy names, required regions, and forbidden region mismatches.
- Include `semanticOwnershipRules` that separate Global Shell, Page Body, Overlay/State, and Mock Data ownership so downstream skills do not duplicate app-level navigation, current-page nav links, global footer, or global brand elements as page-body content.
- Include `duplicatePrevention` for duplicate visible text, duplicate primary actions, duplicate navigation links, duplicate helper links, and repeated state-only content. The rule may allow intentional repeated text only when roles differ.
- Include Figma node hints such as preferred auto-layout direction, table fallback, card grouping, and naming conventions.
- Keep rules aligned with `design-spec.md` section `10.AI页面生成约束`.

### `consistency-rules.json`

Cross-page consistency rules for `pm-06`, `pm-07`, `pm-08`, and `pm-09`.

Required top-level keys:

```json
{
  "meta": {},
  "profileHierarchy": {},
  "profileResolution": {},
  "fingerprintRules": {},
  "lockedProperties": {},
  "layoutBudgetRules": {},
  "semanticUniquenessRules": {},
  "allowedOverrides": {},
  "driftPolicy": {},
  "qaRules": {}
}
```

Rules:

- `profileHierarchy` must define priority: design spec tokens < layout profile < page-intent profile < state-group profile < page-specific content.
- `profileResolution` must define paths under `product/style-profiles/<layout-key>/`.
- `fingerprintRules` must define stable hashes for layout, intent, and state group profiles.
- `lockedProperties` must list properties that cannot drift within the same scope, such as canvas size, shell dimensions, page padding, card width, tabs gap, component heights, shadow, typography scale, and responsive strategy.
- `layoutBudgetRules` must define how much vertical and horizontal space each layout/page-intent/stateGroup profile may consume before it must compress, scroll, or fail. Include safe-area calculations that reserve shell regions and avoid footer/header collision.
- `semanticUniquenessRules` must define duplicate-action and duplicate-region policy across same page body, same page intent, and same state group. These rules must distinguish allowed content text repetition from unsafe duplicated primary actions.
- `allowedOverrides` must distinguish content differences from style differences. Page-specific text, active tab, field labels, table columns, and business data are allowed; shared layout and visual decisions are not.
- `driftPolicy` must require downstream skills to warn or fail when generated JSON or Figma output diverges from a locked profile without explicit user instruction.
- `qaRules` must define consistency scoring and required report fields, including duplicate action audit, layout budget audit, shell collision audit, and profile-fingerprint audit.

### `layout-profile-schema.json`

Schema for project-level profile files created by `pm-06` and orchestrated by `pm-07`.

Required top-level keys:

```json
{
  "meta": {},
  "layoutProfile": {},
  "intentProfile": {},
  "stateGroupProfile": {},
  "layoutBudgetSchema": {},
  "slotContractSchema": {},
  "fingerprintSchema": {},
  "mergeRules": {},
  "qaRules": {}
}
```

Rules:

- `layoutProfile` must support canvas, shell, navigation, page container, spacing, breakpoint, and responsive fallback fields.
- `intentProfile` must support generic page intents such as `auth`, `list-management`, `detail`, `form-create-edit`, `dashboard`, `wizard-flow`, and `checkout-payment`.
- `stateGroupProfile` must support shared state-group bones, such as common card width, method tabs, form control width, button height, QR size, status header, and shared timeline/table geometry.
- `layoutBudgetSchema` must support available canvas height, reserved shell height, content safe area, max root/content/card heights, compression order, scroll fallback, and failure thresholds.
- `slotContractSchema` must support fixed/shared slots, variable slots, allowed component families, min/max slot size, default-visible requirement, and state-specific slot differences.
- `fingerprintSchema` must define canonical JSON fields used to compute `layoutHash`, `intentHash`, and `stateGroupHash`.
- `mergeRules` must define deterministic conflict resolution: stateGroup profile may refine intent profile, but it must not mutate layout shell; page content may fill slots, but it must not alter locked profile geometry.

### `page-intent-patterns.json`

Reusable visual patterns for page intent profiles.

Required top-level keys:

```json
{
  "meta": {},
  "patterns": {},
  "requiredRegions": {},
  "forbiddenRegions": {},
  "defaultSlots": {},
  "layoutBudgets": {},
  "slotContracts": {},
  "duplicatePolicies": {},
  "responsiveRules": {}
}
```

Rules:

- Patterns must be product-agnostic and keyed by generic `pageIntent` names.
- Each pattern must define shared structure, key regions, component families, default slot names, layout direction, spacing references, and responsive behavior.
- Each pattern must define a layout budget: safe content area, max primary panel height, density fallback order, internal scroll candidates, and fail conditions when content cannot fit.
- Each pattern must define slot contracts: which slots are shared across the intent, which slots may vary by page/state, and which component families may fill those slots.
- Each pattern must define duplicate policies for visible page-body regions and primary actions. The policy must be generic, not tied to a project name.
- `auth` must define shared auth page structure without forcing every auth state to use identical content. Example: shell + auth panel + optional brand panel + auth card + method switch + credential/SSO slot + agreement + auxiliary links.
- `list-management` must define page header, filter bar, toolbar, data collection, row action, and pagination slots.
- `detail` must define summary, description fields, related records, timeline/log, and action group slots.
- Patterns must not mention current project page names.

## Update Rules

When updating an existing design spec:

1. Read the existing `design-spec.md` and `index.html`.
2. Read existing `tokens.json`, `figma-tokens.json`, `component-rules.json`, `component-blueprints.json`, `figma-render-rules.json`, and `ai-generation-rules.json` when present.
3. Read existing `consistency-rules.json`, `layout-profile-schema.json`, and `page-intent-patterns.json` when present.
4. Preserve stable design decisions unless the user requests a style change.
5. Preserve user-authored notes and assumptions where still valid.
6. Synchronize any changed tokens across Markdown, HTML, and all JSON files.
7. If the user's request conflicts with an existing rule, apply the newer user instruction and record the change in `0.文档状态`.
